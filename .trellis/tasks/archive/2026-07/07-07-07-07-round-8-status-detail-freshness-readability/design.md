# Status Detail Freshness Readability Design

## Scope

Target files:

- `src/data/siteStatusView.ts`
- `src/pages/SiteStatusDetailPage.tsx`
- `src/styles/flow-pages.css`
- `scripts/check-ui.mjs`
- optionally `.trellis/spec/frontend/quality-guidelines.md` if the UI contract becomes reusable.

## Design Direction

Keep status JSON stable. The previous slice already writes evidence text like:

```text
证据时间：2026-07-06T19:38:47.402Z；证据新鲜度：新鲜（3 小时前）
```

Add a small frontend helper that parses this public-safe evidence phrase into:

- `checkedAt`
- `freshnessLabel`
- `ageText`
- `tone`

The helper should return `null` when the evidence does not include synthetic freshness context. The detail page then renders one extra fact card for synthetic evidence:

- `证据时间`: formatted local date/time when parseable.
- `新鲜度`: a badge plus age text.

Tone mapping:

- `新鲜` -> online tone
- `接近过期` -> degraded tone
- `已过期` -> degraded/offline-adjacent tone, but not a project outage by itself
- `未知` -> unchecked tone

## Compatibility

- Existing `ReliabilityCheck` remains unchanged.
- Static/planned checks continue to display the existing evidence paragraph only.
- The detail route still uses the generated `status.reliabilityProjects` payload from `useSiteStatus()`.

## Safety

- Parse only the known low-sensitive Chinese markers already generated by `site:status`.
- Do not parse or expose URLs, request bodies, provider diagnostics, or raw synthetic snapshot internals.
- Do not run model or credential checks.

## Validation

- `npm.cmd run status:contract`
- `npm.cmd run check:ui`
- `npm.cmd run lint`
- `npm.cmd run build`
- `git diff --check`
- Targeted sensitive scan over changed files.
