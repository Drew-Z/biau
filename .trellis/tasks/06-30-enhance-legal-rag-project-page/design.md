## Scope

This child task enhances the Legal RAG project page in BIAU Port. It does not edit the source Legal RAG repository, rewrite blog posts, or add private knowledge ingestion. The output should be a visitor-readable technical case study backed by multi-source evidence.

1. Product problem and demo value.
2. What the user can do in the workbench.
3. Architecture and RAG pipeline.

6. Current limitations and future optimization direction.

## Data Model

Prefer a reusable project-detail content model over a Legal-RAG-only hard-coded section.

Candidate shape in `src/data/portfolio.ts`:

```ts
export interface ProjectDetailSection {
  title: string
  body?: string
  items?: string[]
  links?: ProjectLink[]
}

export interface ProjectDetailContent {
  overview?: ProjectDetailSection[]
  architecture?: ProjectDetailSection[]
  proof?: ProjectDetailSection[]
  limitations?: ProjectDetailSection[]
  roadmap?: ProjectDetailSection[]
}
```

Then add an optional field to `Project`:

```ts
detailContent?: ProjectDetailContent
```

The exact naming can change during implementation, but the model should support future child tasks for Xunqiu, ERP, games, and pet without duplicating bespoke JSX blocks.

## Rendering

`ProjectDetailPage.tsx` should render generic `project.detailContent` sections when present. Existing custom `OzonErpProjectArticle` and `XunqiuProjectArticle` can remain for now unless adapting the shared renderer is low-risk.

Recommended rendering:

- Keep existing header, hero image, highlights, stack, and links.
- Add a reusable `ProjectDetailContentSections` helper inside `ProjectDetailPage.tsx` or a small component if the file becomes unwieldy.
- Use existing `detail-body`, `detail-block`, `detail-block-wide`, `detail-highlights`, `link-badge`, and `blog-post-body-text` classes where possible.
- Add minimal CSS only for reusable project-case-study sections.

## Assistant Knowledge

Generated project knowledge currently uses only `project.summary` and tags. For Legal RAG, the assistant should gain better context from the enhanced detail data.

Design options:

- Simple option: include flattened `detailContent` body/items in the generated project summary for `server/data/public-knowledge.json`.
- Safer option: add an `assistantSummary` or `assistantContext` field to `Project` and generate knowledge from that field when present.

Recommended: add a typed `assistantContext?: string[]` or `assistantSummary?: string` if flattened UI content would be too long or too presentation-specific. Keep the generated knowledge item shape backward-compatible with `{ id, title, summary, href, tags, visibility }`.

Frontend local fallback in `src/data/assistant.ts` should mirror the same project knowledge logic or reuse a shared helper if practical.
